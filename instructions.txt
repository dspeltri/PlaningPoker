1. Purpose
A lightweight, real‑time Planning Poker web app for agile estimation:

Supports multiple teams running sessions in parallel via rooms.
Each room has:
Name + avatar selection per participant.
A central “table” with participants arranged around it.
A shared backlog (list of items) and current item.
Individual votes (cards) per participant, with Reveal/Reset.
All updates are synchronized in real time across all clients in the same room.
It’s designed to be easy to run on a single Node process (e.g., developer machine or a small internal server).

2. High‑Level Architecture
Backend: Node.js + Express + Socket.IO

Single process, long‑running HTTP + WebSocket server.
Serves static assets and manages per‑room state.
Persists room state to a JSON file on disk (rooms.json) so rooms and backlog survive restarts.
Frontend: Single-page HTML/JS/CSS

No framework; pure DOM manipulation.
Socket.IO client for real‑time events.
Uses URL hash (/#room-XXXXXX) as room ID.
Minimal Bootstrap (CSS) just for buttons/dialogs.
All logic is self-contained in two files:

server.js
public/index.html
3. Data Model
3.1 Rooms (on the server)
Each room is keyed by roomId (the hash string). Shape:

ts

Copy code
type VoteInfo = {
  value: string;    // e.g., "5", "13", "?"
  avatar: string;   // URL path to avatar image
};

type BacklogState = {
  items: string[];      // raw item strings (e.g., "US-123 - Login page")
  currentIndex: number; // index into items, -1 if none
};

type RoomState = {
  revealed: boolean;                    // votes revealed or not
  votes: Record<string, VoteInfo>;      // name → { value, avatar }
  backlog: BacklogState;
};

type Rooms = Record<string, RoomState>;

This rooms object is loaded from disk at startup and saved on any mutation to rooms.json.

4. Backend: server.js
4.1 Technologies
express to serve static files from /public.
http + socket.io for WebSocket communication.
fs/path for JSON file persistence.
4.2 Core responsibilities
Static file serving

js

Copy code
app.use(express.static("public"));

Room state management

loadRoomsFromDisk(): reads rooms.json if present, else returns {}.
saveRoomsToDisk(): writes current rooms to rooms.json (sync I/O).
getRoom(roomId): ensures a RoomState for a given roomId.
Socket.IO events

Within io.on("connection", socket => { … }), per connection we track:

js

Copy code
let currentRoom = null;
let currentName = null;
let currentAvatar = "/avatars/avatar1.jpg";

Handlers:

joinRoom({ roomId, name, avatar })

Leaves previous room if any.
Sets currentRoom, currentName, currentAvatar.
Joins Socket.IO room.
Sends stateUpdate with full RoomState.
Sends backlogUpdate with room.backlog.
updateName({ name, avatar })

Updates currentName / currentAvatar on the server.
vote(value)

Sets room.votes[currentName] = { value, avatar: currentAvatar }.
saveRoomsToDisk().
Emits stateUpdate to the room.
reveal()

Sets room.revealed = true.
saveRoomsToDisk().
Emits stateUpdate.
reset()

Sets room.revealed = false, clears room.votes.
saveRoomsToDisk().
Emits stateUpdate.
setBacklog({ items, currentIndex })

Sets room.backlog.items and currentIndex.
saveRoomsToDisk().
Emits backlogUpdate to the room.
setBacklogIndex(index)

Validates index.
Sets room.backlog.currentIndex.
Clears votes and revealed (new item means new round).
saveRoomsToDisk().
Emits stateUpdate and backlogUpdate.
disconnect

Removes currentName from room.votes if present.
saveRoomsToDisk().
Emits updated stateUpdate.
Listening

js

Copy code
const PORT = process.env.PORT || 3000;
server.listen(PORT, () => {
  console.log(`Planning Poker running at http://localhost:${PORT}`);
});

5. Frontend: public/index.html
A single file with:

All layout and styling in <style>.
All logic in a <script> at the bottom.
5.1 Room selection (lobby)
On first load at / (no hash): user sees a “Select a Room” overlay.
Buttons:
Create New Room:
Generates a random room ID like room-AB12CD.
Sets window.location.hash = "#room-AB12CD" and reloads.
Join Room:
Accepts a full URL (https://…#room-AB12CD) or just the code (room-AB12CD).
Sets hash & reloads.
After reload, ROOM_ID is derived from the hash and the app:

Hides lobby overlay.
Shows the name+avatar dialog.
5.2 Name + avatar dialog
Prompts the user for:
Name (free text).
Avatar (one of 6 local image files under /public/avatars).
On Continue:

Sets read-only name at the top.

Emits:

js

Copy code
socket.emit("joinRoom", {
  roomId: ROOM_ID,
  name: finalName,
  avatar: getMyAvatarUrl(),
});
socket.emit("updateName", { name: finalName, avatar: getMyAvatarUrl() });

Hides the dialog; user can now vote.

5.3 Backlog panel
Right side of the layout:

textarea#itemsInput for “Items (one per line)”.
Add Items button:
Splits textarea into lines → items[].
Emits setBacklog({ items, currentIndex: 0 }).
A list of backlog tiles (backlog-list):
Each tile shows:
Title (the line text).
“Item X of N”.
Clicking a tile calls setBacklogIndex(index) on the server.
Previous/Next buttons:
Also call setBacklogIndex(currentIndex ± 1).
Backlog state is driven by the server’s backlogUpdate and stateUpdate events; the client keeps a local mirror items and currentItemIndex.

5.4 Participants & seats
Left panel: participants as tiles, showing:

Avatar image.
Name.
Status (“Voted ✓” vs “Waiting…” before Reveal; actual value after Reveal).
Center panel: a circular “table” with seats around it:

For each entry in currentState.votes (name → { value, avatar }):
Compute angle around circle.
Draw a “seat card” with:
Avatar.
Name.
Value:
If not revealed: shows “Voted” or “–” and greys it out.
If revealed: shows actual value.
5.5 Cards & voting
Bottom row: clickable .vote-card elements from FIB_CARDS.
On click:

Locally mark card as selected (myCurrentVote).
Emit vote(value).
The server updates room.votes and broadcasts stateUpdate, which updates participants and seats.

5.6 Reveal & Reset
Reveal button:
Calls socket.emit("reveal").
Server marks revealed = true, broadcasts stateUpdate.
Reset button:
Calls socket.emit("reset").
Server clears votes and sets revealed = false, broadcasts stateUpdate.
5.7 Summary
Under the table:

updateStatusAndSummary() calculates:
Before Reveal: how many have voted (votedCount) vs totalVotes.
After Reveal: min/max/avg of numeric votes.
Non-numeric values (like “?”) are ignored for stats.
Shows avg in the center of the table as tableAverage.
5.8 Copy link button
In the header:

Shows current room label (Room: room-AB12CD).
Copy link button:
If ROOM_ID is set:
Uses navigator.clipboard.writeText(window.location.href).
Shows status “Room link copied to clipboard.”
If no room: shows “No room selected yet to copy.”

7. Notes for Extension
A staff dev could easily extend this with:

Room metadata (createdAt, friendly name, lastActivity).
Room pruning (remove stale rooms from rooms.json).
Per‑user avatars stored server-side instead of inferred per client.
Security:
Room passwords.
IP allowlists or reverse-proxy auth.
Export:
Endpoint to download votes/backlog as CSV/JSON per room.
The current codebase is small and intentionally simple; changes are localized primarily to:

server.js: room state and events.
The <script> inside public/index.html: front-end state and UI wiring.
